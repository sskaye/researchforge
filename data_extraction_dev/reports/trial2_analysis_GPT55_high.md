# Trial 2 GPT-5.5 High Failure Analysis

This is a diagnosis of my `Trial2-full-gpt55_high` run against the 168-paper corpus, based on the comparison report, the skill text, the extraction prompt template, and the artifacts left in the trial directory. I am not re-running extraction here.

## Summary

The run did follow the most important v1.4 constraint better than the prior failed GPT-5.5 run: I did not build a Python regex extractor that bulk-generated rows from `pdftotext`. The rows were produced through LLM batch extraction, with deterministic scripts used for orchestration, merging, duplicate cleanup, schema checks, quote-presence checks, and a limited Phase 4 audit.

The residual failures came from a different gap: I let "row exists in paper text" and "worker says it read the paper" substitute for row-level evidence support. For some rows, the final CSV preserved partial compound-name fragments or masthead/header text as `evidence_quote`. That means the mandatory quote re-confirmation and full-name reassembly protocol was not actually enforced on every committed row.

The comparison report's audit classified my random-100 result as 86% pass, with 14 failures:

- 7 `flagged_compound_name_truncated`
- 4 `flagged_evidence_quote_not_found`
- 2 `flagged_compound_mismatch`
- 1 `flagged_value_mismatch`

At the field level, the report categorized this as 9 compound-name errors, 1 value/property error, 0 source/DOI errors, and 4 quote-only failures. Importantly, the report says those 4 quote-only rows were not merely correct measurements with bad quote spans: the quote was masthead text and the row lacked actual mp/bp measurement support in the paper.

## 1. Root Cause From My Side

### What I was actually doing

I used a mixed workflow:

1. I read the skill and set up a batch workflow over `/mp_bp_full_set`.
2. I generated a manifest and batch CSVs to split the corpus.
3. I used parallel LLM worker/subagent extraction over paper batches. Those workers were supposed to read the papers directly and write rows by the skill protocol.
4. I used helper scripts after extraction to merge batch CSVs, renumber rows, deduplicate, run deterministic checks, prepare temporary text/index material for collection-style paper directories, select Phase 4 samples, and apply Phase 4 corrections.
5. I ran an internal Phase 4 audit on a limited sample, fixed the rows that sample caught, and delivered the output with most rows still marked `pending_verification`.

The helper scripts did not serve as a forbidden regex extractor in the Trial-1 sense. They did not bulk scan all paper text for mp/bp patterns and directly generate the dataset. They did, however, materially shaped the output:

- corpus enumeration and batch assignment
- final row ordering and IDs
- duplicate removal and duplicate-report generation
- temporary paper-root/index construction so quote checkers could see paired HTML/PDF collection papers
- sample selection for internal Phase 4
- systematic corrections from the internal sample

The actual row contents were generated by the LLM extraction workers and the later gap-pass extraction, not by a single deterministic extraction script.

### What I did not do

I did not personally re-read each paper for each final row before delivery. I relied on batch workers, deterministic quote-presence/schema checks, and the limited Phase 4 sample. This is the main operational failure.

The failing rows show that at least some row construction was effectively based on partial text windows, search results, or copied snippets that were not re-expanded and re-checked against the actual measurement context. The evidence is in the failure shapes:

- `...Carboxamid`, `...Carbo`, `8-(1-(3-(5'-`, and `Phenyl(6'-` are not chemically meaningful final names; they are line/window/field fragments.
- Masthead-as-quote rows could pass a naive "quote appears somewhere in the paper" check while still not proving the measurement.
- The 4i/4f and substituent-position errors are classic adjacent-row or label-binding mistakes: the measurement area was nearby, but the row-level binding was not independently confirmed.
- The decomposition/property error (`mp. 43-48 deg C` recorded as decomposition) shows the property field was inferred loosely rather than constrained by the quote text.

So the best reconstruction is: the run was not a regex-extractor run, but it was also not a fully row-by-row evidence-locked read. It was an LLM-batch extraction run with scripts used around the extraction and with insufficient enforcement of the row-level quote/compound/value binding step.

### Phase 4

I did run Phase 4, but only as an internal sample audit, not as exhaustive verification of all 907 rows.

The trial directory contains `phase4_audit_summary.json` and verdict files. The README records 40 sampled rows, 34 initially passing, and 6 requiring correction/removal. I corrected those specific rows and some adjacent/systematic patterns. The final statuses still show most rows as `pending_verification`.

That was a partial verification step. It improved the output, but it did not justify treating the entire dataset as fully verified. The external random-100 audit later found the residual categories above.

## 2. Where The Skill Failed To Constrain Me

The skill contained the right principles, but several parts left enough interpretation room for my workflow to pass through.

### Direct reading vs. delegated batch reading

The skill says:

> **This skill is applied by an LLM agent reading each paper directly. It is NOT applied by writing a Python extraction script.**

It also says:

> **For each paper, write extracted rows yourself.** Apply the EXTRACTION_PROMPT_TEMPLATES.md Template 1 protocol per paper. Each row must include a verbatim `evidence_quote` drawn from text you actually read.

And:

> **Run extraction as a single batch script.** Parallelization is fine - you can dispatch multiple LLM agent invocations to handle different paper batches concurrently - but each agent must read its papers directly, not write a regex pipeline.

I interpreted this as permission to use worker agents for separate paper batches and then merge their outputs, as long as no deterministic regex pipeline generated the rows. That interpretation was not unreasonable, but it created an accountability gap: the coordinator did not have to prove that each committed row had been personally re-confirmed from source text after merge.

The skill should constrain not only the extraction engine, but also the coordinator behavior after parallel extraction. In this run, the workers may have read papers directly, but the final deliverable still contained rows that no final pass had evidence-locked.

### Search as navigation vs. search as extraction

The template says:

> **Step 3 - Find candidate locations.**

The surrounding protocol allows scanning for candidate locations and then extracting from those locations. That is necessary. But my failure mode sits in the gray area between "search found the place to read" and "search/snippet became the row." The skill forbids a regex pipeline, but it does not explicitly forbid committing from a partial search snippet, truncated text window, or first matching header/title block.

The masthead rows are the clearest example. The extractor evidently found text in the paper file, but not the measurement-bearing span. A deterministic quote-presence check was satisfied while the scientific evidence requirement was not.

### Quote re-confirmation was textually strong but mechanically weak

The skill says:

> **Quote re-confirmation (mandatory before committing each row).** After you've drafted the row, BEFORE adding it to the output:

and:

> Do a substring search for your `evidence_quote`. It must be present **verbatim** ...

The prompt template makes it even stronger:

> **Step 6 - Quote re-confirmation. THIS IS MANDATORY AND NON-NEGOTIABLE.**

and:

> If the substring isn't found, your quote is wrong. Rewrite it from the actual paper text or DROP the row.

I treated the deterministic quote checker as a proxy for part of this obligation. But "substring found" is only one part of evidence quality. The failing masthead rows likely had substrings that existed in the file, just not substrings containing a measurement. The checker did not require the quote to contain `value_raw`, a property token, or a compound/code token. This let a header block pass as evidence.

### Truncation rule was present but not broad enough to become a hard gate

The skill says:

> **Drop these aggressively** - they are common failure modes:

and includes:

> Compound names that are obviously truncated mid-token.

The prompt template says:

> `compound_name`: full IUPAC, common, or trivial name as printed. **Never truncated.** For multi-line PDF names that wrap, reassemble them into a single string.

It also says:

> **Step 7.5 - Multi-line compound name reassembly (mandatory when source is a wrapped PDF).**

and:

> When you're not sure, **drop the row rather than commit a truncated name**.

I evidently did not apply that rule reliably at merge/finalization time. The language is clear, but it depends on the extractor noticing truncation. The examples in the skill focused on common PDF-wrap and locant-prefix cases; my failures included arbitrary terminal fragments (`Carboxamid`, `Carbo`), dangling ring/quote fragments (`8-(1-(3-(5'-`), and even a paragraph copied into `compound_name`. There was no deterministic shape lint that would have made these impossible to deliver.

### Phase 4 scope was ambiguous enough to allow a limited sample

The checklist says:

> I will run Phase 4 independent verification (fresh-context agent re-audit of a random sample) before declaring success.

The main workflow says:

> **Run mandatory Phase 4 independent verification** before declaring the output ready. Dispatching fresh Claude agents (or equivalent) to re-audit a random sample against the source papers is required, not optional.

Later, the Phase 4 section says:

> A different agent ... verifies each row

I interpreted the mandatory Phase 4 requirement as satisfied by a random sample, because the preflight and workflow wording explicitly mention a random sample. The later "each row" wording points in a stricter direction, but the skill does not define when sample verification is enough, what sample size is required, or what to do when a sample finds failures.

In this run, a 40-row sample caught real defects, I fixed those, and then stopped. The skill did not force a second sample, an expanded audit, or a full audit after nonzero sample failures.

### "Fix flagged rows" did not address unflagged residual risk

The skill says:

> If any rows are flagged, **fix them before delivery**. Do not deliver flagged rows mixed with verified ones - they should sit in `flagged_review` status with notes.

I complied for rows that were flagged in my internal sample. But that instruction only covers known flagged rows. It does not say that sample-discovered failures require a systematic sweep over all rows with the same source, compound-name shape, quote pattern, or property context. I did perform a few adjacent/systematic fixes, but not enough to catch the failure classes later found by the external audit.

## 3. Concrete Skill Changes I Would Suggest

### Add a coordinator accountability rule after the parallelization paragraph

Suggested SKILL.md insertion after the existing parallelization sentence:

> If you use parallel LLM workers, the coordinator remains responsible for row-level evidence support in the merged CSV. Before delivery, run the row-support checks on the merged file and manually inspect every row those checks flag. Do not treat worker summaries, successful CSV merging, or quote-substring presence as proof that a row is evidence-locked.

This preserves the useful parallel workflow but closes the gap where the final agent can rely on worker outputs without verifying merged-row integrity.

### Add an explicit anti-pattern for snippet/header extraction

Suggested anti-pattern entry near the "Drop these aggressively" list:

> Header, masthead, title-block, author-list, abstract-metadata, or journal-name text used as `evidence_quote`. A valid `evidence_quote` must be the local measurement-bearing span: it must contain the reported value or table-cell value and enough compound/code context to bind that value to the row. A quote that appears verbatim in the paper is still invalid if it does not support the property/value.

This would directly target the masthead-as-quote failures.

### Add `scripts/quote_support_lint.py`

New deterministic check:

`scripts/quote_support_lint.py --csv extracted.csv --papers /path/to/papers`

For each row:

- Normalize `evidence_quote`, `value_raw`, and whitespace.
- Require `evidence_quote` to contain the numeric value or range from `value_raw` after normalization, unless the row explicitly cites a table location and the quote is a complete copied table row.
- Require the quote to contain either a compound code/name token from `compound_name` or an explicit table row label that appears in `compound_name`/`notes`.
- Reject likely masthead/front-matter quotes: repeated journal title strings, DOI/title/author blocks, publisher boilerplate, or first-page metadata spans without an mp/bp/decomp/DSC keyword or value.
- For `property=decomposition`, require `dec`, `decomp`, `decomposition`, `Td`, `TGA`, or equivalent in the quote/location/notes. For `melting_point`, reject quotes that only support decomposition unless `data_type` explicitly captures onset/decomposition.

This catches my masthead rows and the decomposition-vs-mp property mismatch before Phase 4.

### Add or expand `scripts/compound_name_shape_lint.py`

New deterministic check:

`scripts/compound_name_shape_lint.py --csv extracted.csv`

Flag rows where `compound_name` has any of these shapes:

- Ends in obvious unfinished fragments: `Carboxamid`, `Carbo`, `carbonitril`, `sulfonamid`, a dangling hyphen, or a dangling apostrophe/prime.
- Has unbalanced parentheses, brackets, braces, or quote/prime-like delimiters.
- Ends immediately after an opening scaffold fragment such as `Phenyl(`, `8-(1-(`, `6'-`, `5'-`, or similar dangling locants.
- Is implausibly long and contains procedure words such as `solution`, `added`, `filtered`, `yield`, `afforded`, `NMR`, `IR`, or `mp`.
- Contains multiple sentence-like clauses or semicolons typical of an experimental paragraph rather than a compound name.

The check should not auto-delete rows. It should fail the run and require manual review. This would have caught the 7 truncated-name failures and the paragraph-as-compound row.

### Strengthen Step 6 from "quote found" to "quote supports"

Suggested prompt-template replacement under Step 6:

> Substring presence is necessary but not sufficient. After confirming the quote is verbatim, confirm the quote itself supports the row: it must contain the reported value/range/relation and local compound/code context. If the value, compound/code, or property must be inferred from text outside the quote, expand the quote to the full local span or drop the row.

This prevents an extractor from satisfying the literal substring test with a non-evidentiary quote.

### Add a "no snippet commit" rule

Suggested template sentence under Step 3:

> Search tools may locate candidate regions only. Never commit a row from a search-result line, fixed-width snippet, PDF text fragment, or first match alone. Open/read enough surrounding text to see the table heading, row label, neighboring rows, and the measurement column before writing the row.

This targets the search/snippet failure without forbidding useful `rg`/`pdftotext` navigation.

### Clarify Phase 4 escalation after failures

Suggested SKILL.md change:

> Minimum Phase 4 for corpus runs is a random sample of at least 100 rows or 10% of rows, whichever is smaller, with source-paper re-audit by fresh context. If the sample has any failures, perform a targeted sweep for the same failure class across the full CSV, fix/drop those rows, and run a second fresh random sample. Do not declare the output ready after a failing sample plus only local corrections.

For small outputs, the skill could require all-row Phase 4. For large outputs, this gives a concrete standard and an escalation rule.

### Add status semantics for sampled vs. fully verified rows

Suggested SKILL.md addition:

> `verified_extraction` may only be used for rows independently checked against the paper. Rows that pass deterministic checks but were not Phase-4 audited must remain `pending_verification`. The final README must report counts for verified, corrected, removed, and unaudited rows.

I mostly did this in the README, but making it explicit helps prevent "Phase 4 ran" from sounding like "the dataset is verified."

### Add a near-duplicate/adjacent-row binding check

New deterministic check:

`scripts/adjacent_binding_lint.py --csv extracted.csv`

Flag clusters with same source, same property, same value/range, and highly similar compound names or compound codes, especially where only the code differs. This would not prove an error, but it would have surfaced the 4i/4f-style mismatch for human review.

## Bottom Line

The skill successfully prevented the old catastrophic pattern: I did not write a bulk regex extractor, and the run improved substantially. The remaining failures came from weaker enforcement at the row-support layer. My process allowed LLM-generated batch rows to survive if they were schema-valid and had a quote substring somewhere in the paper, even when the compound name was visibly truncated or the quote did not contain the measurement.

The most useful fixes are therefore not more reminders that regex extraction is forbidden. They are concrete row-support gates: quote must contain/support the value, compound names must pass shape lint, search snippets cannot be committed as rows, and a failing Phase 4 sample must trigger a second sample plus systematic sweeps.
